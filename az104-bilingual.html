1
Tenant — o que é e por que é a base de tudoTenant — what it is and why it is the foundation of everythingfundaçãofoundation▾

O que é: um tenant é uma instância dedicada e isolada do Microsoft Entra ID que sua organização recebe ao assinar um serviço Microsoft (Azure, Microsoft 365). É o "diretório" onde vivem todas as suas identidades — usuários, grupos, apps.

  • Cada tenant tem um domínio inicial tipo empresa.onmicrosoft.com (você pode adicionar domínios customizados depois)
  • Tem um Tenant ID — um GUID único que identifica seu diretório
  • É um limite de segurança e identidade: usuários de um tenant não existem em outro por padrão
  • Uma organização normalmente tem um tenant, mas pode ter vários (ex: fusões, separação prod/test)
  • Toda subscription do Azure é associada a exatamente um tenant para autenticação
O tenant é o prédio da sua empresa com sua própria portaria e lista de funcionários. Ninguém de outro prédio entra sem ser convidado (Guest). As subscriptions são os contratos de serviço que esse prédio assina.
Tenant ≠ Subscription. Tenant é o diretório de identidades (Entra ID). Subscription é o contrato de billing dos serviços. Um tenant pode ter várias subscriptions, mas cada subscription pertence a um único tenant.
2
Microsoft Entra ID — o serviço de identidade (ex-Azure AD)Microsoft Entra ID — the identity service (formerly Azure AD)fundaçãofoundation▾

O que é: o serviço de gerenciamento de identidade e acesso da Microsoft na nuvem. Era chamado de Azure Active Directory (Azure AD) até 2023 — o nome mudou para Microsoft Entra ID, mas a função é a mesma. É onde você gerencia quem pode acessar o quê.

  • Autenticação (AuthN): confirma que você é quem diz ser — login, MFA
  • Autorização (AuthZ): define o que você pode fazer — trabalha junto com RBAC
  • Gerencia usuários, grupos, dispositivos, aplicações e service principals
  • Base para SSO (Single Sign-On) entre Azure, Microsoft 365 e apps terceiros
  • Suporta identidades híbridas — sincroniza com AD on-premises via Entra Connect
  • Não é o mesmo que Active Directory (AD DS) on-premises — Entra ID é baseado em nuvem, usa protocolos web (OAuth, SAML, OpenID Connect), não LDAP/Kerberos
Entra ID é a portaria digital da empresa. Ele verifica o crachá (autenticação), consulta a lista do que você pode acessar (autorização) e libera as portas certas — tudo pela nuvem, não por servidor físico.
Entra ID (nuvem) ≠ Active Directory Domain Services (on-premises). AD DS usa LDAP/Kerberos e roda em servidores da empresa. Entra ID usa protocolos web e roda na nuvem Microsoft. Questão que mencionar "Azure AD" está falando de Entra ID — é o mesmo serviço, nome antigo.
3
Users & Groups — tipos e diferençasUsers & Groups — types and differencesdecorarmemorize▾

Toda identidade humana no Entra ID é um user, e usuários são organizados em groups para facilitar a atribuição de acesso em massa.

  • Member user: conta interna da organização (funcionário)
  • Guest user (B2B): conta externa convidada — parceiro, fornecedor; usa a identidade da própria empresa dele
  • Security group: usado para atribuir RBAC, Policy e acesso a recursos
  • Microsoft 365 group: usado para colaboração (email compartilhado, Teams, SharePoint)
  • Assigned membership: você adiciona/remove membros manualmente
  • Dynamic membership: membros entram/saem automaticamente por regra de atributo — exige licença P1
  • UPN (User Principal Name): o login, tipo ana@empresa.com — pode mudar
  • Object ID: GUID imutável do usuário — use em scripts, nunca muda
Atribuir acesso a grupos em vez de usuários individuais é como dar chaves a cargos, não a pessoas. Quando alguém troca de função, você só muda o grupo — não precisa refazer 20 permissões.
Dynamic Groups exigem Entra ID P1. Você NÃO pode adicionar membros manualmente em um dynamic group — eles entram por regra de atributo. Questão "automaticamente adicionar novos funcionários do departamento X" → Dynamic Security Group + P1.
4
Licenças Entra ID — Free, P1 e P2Entra ID Licenses — Free, P1 and P2decorarmemorize▾

Muitas features importantes só existem em certas licenças. Saber qual feature precisa de qual licença é decisivo.

  • Free: usuários, grupos, SSO básico, MFA de segurança básica
  • P1: Conditional Access, Dynamic Groups, self-service password reset com writeback, gerenciamento avançado de grupos
  • P2: tudo do P1 + PIM (Privileged Identity Management), Identity Protection (detecção de risco), access reviews
As licenças são planos de assinatura crescentes. Free é o básico que todo mundo tem, P1 adiciona controle de acesso condicional, P2 adiciona segurança avançada e acesso just-in-time (PIM).
Conditional Access e Dynamic Groups → P1. PIM e Identity Protection → P2. Se a questão pede acesso "just-in-time" a roles privilegiadas → PIM → licença P2.
5
MFA & Conditional Access — segurança de acessoMFA & Conditional Access — access securitynovonew▾

MFA (Multi-Factor Authentication): exige um segundo fator além da senha (app autenticador, SMS, token). Conditional Access: políticas "se-então" que decidem quando exigir MFA ou bloquear acesso.

  • Conditional Access avalia sinais: usuário, localização, dispositivo, app, risco de login
  • Exemplo de política: "se login vier de fora do Brasil, exigir MFA"
  • Exemplo: "se dispositivo não estiver conforme (não gerenciado), bloquear acesso"
  • Conditional Access exige licença P1
  • Security Defaults: MFA básico gratuito para todos, mas sem granularidade
Conditional Access é o segurança inteligente da portaria: ele não pede documento de todo mundo igual — pede identidade extra (MFA) só de quem chega em horário estranho ou de local desconhecido.
Security Defaults (grátis) é tudo-ou-nada. Conditional Access (P1) permite regras granulares. Questão que pede "exigir MFA só para admins" ou "só de fora da rede corporativa" → Conditional Access → P1.
6
Service Principal & Managed Identity — identidade para appsService Principal & Managed Identity — identity for appsmais cobradomost tested▾

Nem toda identidade é humana. Apps, scripts e serviços também precisam se autenticar — e é aí que entram Service Principals e Managed Identities.

  • Service Principal: a identidade de uma aplicação dentro do tenant. Tem credencial (client secret ou certificado) que você precisa gerenciar e rotacionar
  • Managed Identity System-assigned: criada e vinculada a UM recurso (ex: uma VM). Deletada junto com o recurso. Não pode ser compartilhada
  • Managed Identity User-assigned: criada como recurso independente, pode ser atribuída a vários recursos ao mesmo tempo
  • A grande vantagem da Managed Identity: o Azure gerencia a credencial — sem secret no código, sem rotação manual
  • Managed Identity é, por baixo, um tipo especial de Service Principal gerenciado pela plataforma
Service Principal é um crachá de robô que você mesmo tem que renovar. Managed Identity é o mesmo crachá, mas o próprio prédio renova automaticamente — você nunca lida com a validade.
System-assigned: 1:1 com o recurso, morre com ele. User-assigned: compartilhável entre vários recursos. Questão "múltiplas VMs precisam da mesma identidade para acessar o Key Vault" → User-assigned Managed Identity.

⚠ Resumo — as confusões mais comuns de identidade⚠ Summary — the most common identity confusions

Tenant (diretório) ≠ Subscription (billing). Entra ID (nuvem, web) ≠ AD DS (on-prem, LDAP). Conditional Access/Dynamic Groups → P1. PIM/Identity Protection → P2. Managed Identity sempre que possível — sem secret para gerenciar.

VNet & Subnets — sua rede na nuvemVNet & Subnets — your network in the cloudfundaçãofoundation▾

O que é: a Virtual Network (VNet) é sua rede privada isolada no Azure. Funciona igual à rede do seu datacenter — você define o espaço de endereços (CIDR) e divide em subnets.

  • Address space (CIDR): o intervalo de IPs da VNet, ex: 10.0.0.0/16
  • Subnet: subdivisão da VNet, ex: 10.0.1.0/24 para web, 10.0.2.0/24 para banco
  • Address space não pode sobrepor com VNets que serão conectadas (peering/VPN)
  • Azure reserva 5 IPs por subnet (primeiro, último e 3 internos)
  • Subnet delegation: reserva a subnet para um serviço PaaS específico (ex: Azure SQL Managed Instance)
A VNet é o terreno da sua rede e as subnets são os lotes demarcados dentro dele. Você não pode ter dois terrenos com os mesmos endereços se quiser conectá-los por uma ponte (peering).
Azure reserva 5 IPs por subnet — então uma /24 (256 IPs) só tem 251 utilizáveis. Address spaces sobrepostos impedem o peering. Questões de planejamento de IP exploram isso.
2
NSG & ASG — controle de tráfegoNSG & ASG — traffic controlmais cobradomost tested▾

NSG (Network Security Group): firewall básico com regras de permitir/negar tráfego. ASG (Application Security Group): agrupa VMs logicamente para usar nas regras do NSG.

  • Stateful: se você permite tráfego de entrada, a resposta de saída é automática — não precisa regra de retorno
  • Priority: menor número = maior prioridade (100 vence 200). Range 100–4096
  • Regras default: AllowVNetInBound, AllowAzureLoadBalancerInBound, DenyAllInBound (menor prioridade)
  • Associação: NSG pode estar na subnet (afeta todas as VMs) ou na NIC (afeta uma VM)
  • Effective rules: se houver NSG na subnet E na NIC, ambos são avaliados — os dois precisam permitir
  • ASG: em vez de listar IPs nas regras, agrupa NICs — ex: regra "permitir do ASG-Web para o ASG-DB"
NSG é a lista de regras do porteiro (quem entra, quem sai). ASG é dar apelidos aos grupos — em vez de dizer "libere para 10.0.1.4, 10.0.1.5, 10.0.1.6", você diz "libere para o grupo Web".
NSG é stateful — não precisa de regra de retorno. Se houver NSG na subnet e na NIC, o tráfego precisa passar por AMBOS. Menor número de prioridade vence. Questão de troubleshooting: "tráfego bloqueado" geralmente é regra de prioridade menor negando antes.
3
VNet Peering — conectar redesVNet Peering — connecting networkspegadinha clássicaclassic trap▾

O que é: conecta duas VNets para que se comuniquem como se fossem uma só, usando o backbone privado do Azure (sem passar pela internet).

  • Regional peering: VNets na mesma região
  • Global peering: VNets em regiões diferentes — maior latência, cobra egress
  • NÃO é transitivo: se A↔B e B↔C, isso NÃO significa A↔C
  • Para tráfego transitivo (hub-and-spoke): precisa de UDR + Azure Firewall/NVA no hub
  • Peering exige address spaces sem sobreposição
Peering é construir uma ponte direta entre dois terrenos. Só porque o terreno A tem ponte para o B, e o B tem ponte para o C, não existe uma ponte automática de A para C — você teria que construí-la ou passar por um pedágio central (firewall no hub).
VNet Peering NÃO é transitivo. Em hub-and-spoke, Spoke1 não fala com Spoke2 automaticamente. Solução: UDR (User Defined Route) apontando para Azure Firewall/NVA no Hub.
4
Service Endpoint vs Private Endpoint — acesso privado a PaaSService Endpoint vs Private Endpoint — private access to PaaSpegadinha clássicaclassic trap▾

Ambos deixam VMs acessarem serviços PaaS (Storage, SQL) de forma mais segura, mas de jeitos bem diferentes.

  • Service Endpoint: roteia o tráfego da subnet pelo backbone Azure. Mais seguro que internet, MAS o serviço PaaS ainda tem um IP público
  • Private Endpoint: cria uma NIC com IP privado dentro da sua VNet que representa o serviço PaaS. O serviço fica completamente privado
  • Private Endpoint permite acesso de on-premises via VPN/ExpressRoute; Service Endpoint não
  • Private Endpoint funciona entre VNets e regiões; Service Endpoint é preso à subnet
  • Para compliance estrito (nenhum endpoint público) → sempre Private Endpoint
Service Endpoint é uma estrada expressa privada até a loja, mas a loja continua com porta para a rua pública. Private Endpoint traz um balcão da loja para dentro do seu prédio — ninguém de fora nem sabe que ele existe.
Service Endpoint = PaaS mantém IP público. Private Endpoint = IP privado dentro da VNet, serviço 100% privado. Questão com "sem exposição pública" ou "acessível de on-premises" → Private Endpoint.
5
Load Balancing — qual usar em cada cenárioLoad Balancing — which to use in each scenariomuito cobradoheavily tested▾

Escolher o balanceador certo é uma das decisões mais cobradas. A chave é: camada (L4 vs L7), escopo (regional vs global) e se precisa de WAF.

  • Azure Load Balancer: L4 (TCP/UDP), regional. Não entende HTTP, sem WAF. Para VMs e VMSS
  • Application Gateway: L7 (HTTP/HTTPS), regional. WAF, path-based routing, cookie affinity, TLS termination
  • Traffic Manager: baseado em DNS, global. Roteamento por latência/prioridade/peso. Não inspeciona conteúdo
  • Azure Front Door: L7 global, Anycast, CDN integrado, WAF global, failover automático
  • Regra de bolso: HTTP global → Front Door | HTTP regional → App Gateway | TCP/UDP → Load Balancer | DNS global → Traffic Manager
Load Balancer é o porteiro que distribui pessoas entre filas sem olhar o conteúdo. App Gateway é o recepcionista que lê o pedido ("quer suporte? vá para o balcão 3") e ainda barra visitantes suspeitos (WAF).
Se a questão menciona "path-based routing", "WAF" ou "redirecionar /api para outro backend" → Application Gateway (L7). Load Balancer é L4 e não entende HTTP. "Global + CDN + failover" → Front Door.
6
Conectividade híbrida — ligar on-premises ao AzureHybrid connectivity — connecting on-premises to Azurenovonew▾
  • VPN Gateway Site-to-Site (S2S): túnel IPSec sobre internet pública, conecta rede on-prem inteira. SLA 99.9%
  • VPN Point-to-Site (P2S): clientes individuais se conectam (home office). OpenVPN, SSTP ou IKEv2
  • ExpressRoute: conexão privada dedicada, NÃO passa pela internet. Latência baixa, alto throughput. SLA 99.95%. Custo alto
  • ExpressRoute + VPN: ER como primário, VPN como failover — máxima resiliência
  • BGP: obrigatório com ExpressRoute; opcional com VPN S2S. VPN Gateway SKU Basic não suporta BGP
VPN S2S é um túnel seguro construído dentro da estrada pública (internet). ExpressRoute é uma estrada exclusiva sua, que ninguém mais usa — mais rápida e privada, mas cara de construir.
"Latência mínima, sem passar pela internet, compliance" → ExpressRoute. "Custo baixo, conectar filial" → VPN S2S. "Usuários remotos individuais" → VPN P2S. ExpressRoute exige BGP.
7
Azure DNS — resolução de nomesAzure DNS — name resolutionnovonew▾
  • Azure DNS público: hospeda seu domínio público (ex: empresa.com) na infraestrutura Microsoft
  • Private DNS Zone: resolução de nomes DNS interna dentro de VNets, sem expor à internet
  • Auto-registration: Private DNS pode registrar automaticamente os nomes das VMs da VNet
  • Private DNS é essencial para Private Endpoints resolverem para o IP privado correto
Private Endpoint precisa de Private DNS Zone configurada para resolver o FQDN do serviço PaaS para o IP privado. Sem isso, o nome ainda resolve para o IP público. Pegadinha comum em troubleshooting de conectividade.

⚠ Resumo — pegadinhas de networking⚠ Summary — networking traps

VNet Peering não é transitivo (use UDR + Firewall no hub). Service Endpoint mantém IP público, Private Endpoint é 100% privado. NSG é stateful e ambos (subnet+NIC) precisam permitir. L4 (LB) vs L7 (App Gateway/WAF). ExpressRoute para privado sem internet.

Tipos de storage — os 4 serviços de uma Storage AccountStorage types — the 4 services in a Storage Accountfundaçãofoundation▾

Uma Storage Account agrupa 4 tipos de armazenamento, cada um para um propósito.

  • Blob: objetos não estruturados — imagens, vídeos, backups, logs. O mais cobrado
  • File: compartilhamento de arquivos via SMB/NFS — substitui file server on-prem
  • Queue: fila de mensagens para desacoplar aplicações
  • Table: NoSQL simples chave-valor
  • Performance tiers: Standard (HDD, barato) vs Premium (SSD, baixa latência)
A Storage Account é um armazém com quatro áreas: prateleiras para caixas soltas (Blob), arquivo de pastas compartilhadas (File), esteira de pedidos (Queue) e um fichário simples (Table).
Azure Files usa SMB/NFS e pode ser montado como drive de rede. Blob é acessado via REST/HTTP, não é um "drive". Questão "substituir file server on-prem com compartilhamento mapeado" → Azure Files, não Blob.
2
Redundância — LRS, ZRS, GRS e as variantes RARedundancy — LRS, ZRS, GRS and the RA variantsmuito cobradoheavily tested▾

A escolha de redundância é sobre contra o quê você quer se proteger e quanto está disposto a pagar. Este é o tópico #1 de storage na prova.

  • LRS (Local): 3 cópias no mesmo datacenter. Protege contra falha de disco/rack. Mais barato
  • ZRS (Zone): 3 cópias em 3 zonas de disponibilidade diferentes. Protege contra falha de datacenter inteiro (zona)
  • GRS (Geo): LRS + replica assíncrona numa região par (a centenas de km). Protege contra desastre regional. Secundário só acessível após failover
  • GZRS: ZRS + replica em região par. Protege contra zona E região. Melhor proteção
  • RA-GRS / RA-GZRS: o "RA" = Read Access. A região secundária fica disponível para leitura a qualquer momento (sufixo -secondary na URL), sem esperar failover
LRS é guardar 3 cópias do documento na mesma sala. ZRS é espalhar em 3 salas do prédio. GRS é mandar uma cópia para outra cidade. O "RA" é ter a chave da filial de outra cidade sempre no bolso, sem precisar de autorização de emergência.
GRS: secundário só acessível APÓS failover. RA-GRS: leitura do secundário SEMPRE disponível. Questão "ler dados da região secundária sem acionar failover" → RA-GRS ou RA-GZRS.
3
Access tiers — Hot, Cool, Cold e ArchiveAccess tiers — Hot, Cool, Cold and Archivedecorarmemorize▾

Tiers de blob balanceiam custo de armazenamento vs custo de acesso, conforme a frequência de uso do dado.

  • Hot: acesso frequente. Storage caro, operações baratas
  • Cool: acesso esporádico (mínimo 30 dias). Storage mais barato, operações mais caras. Early deletion fee
  • Cold: acesso raro (mínimo 90 dias)
  • Archive: acesso raríssimo. Storage baratíssimo, MAS o dado fica offline — precisa de rehydration (até 15h Standard / ~1h High Priority) antes de ler
  • Lifecycle Management: regras que movem blobs entre tiers automaticamente (ex: mover para Cool após 30 dias sem acesso, para Archive após 180)
Hot é a gaveta da mesa (pega rápido, mas ocupa espaço nobre). Cool é o armário. Archive é o depósito trancado no subsolo — barato para guardar, mas você espera horas para conseguir buscar algo.
Blob em Archive NÃO pode ser lido diretamente — precisa de rehydration (até 15h). Questão com "acesso imediato" ou SLA de minutos → Archive está errado, use Cool/Cold.
4
SAS, Keys & RBAC — controle de acessoSAS, Keys & RBAC — access controlmais cobradomost tested▾
  • Storage Account Key: duas chaves com acesso TOTAL à conta. Trate como root password — quem tem, faz tudo
  • SAS (Shared Access Signature): token com acesso granular e prazo de validade. Tipos: Service SAS, Account SAS
  • User Delegation SAS: o mais seguro — assinado com credencial do Entra ID, não com a account key. Respeita RBAC
  • RBAC (Entra ID): roles como Storage Blob Data Reader/Contributor — separa o controle dos dados do controle da conta
  • Boa prática: preferir Entra ID/RBAC e User Delegation SAS sobre account keys
A account key é a chave-mestra do prédio. SAS é um crachá temporário de visitante (só o 3º andar, válido até as 18h). User Delegation SAS é esse crachá emitido pela portaria oficial (Entra ID), com registro de quem autorizou.
User Delegation SAS usa credencial Entra ID e é mais seguro que SAS baseado em account key. Se a account key vaza, você precisa rotacionar as duas chaves — o que invalida TODOS os SAS baseados nelas.

  • Standard SSD: web servers, produção leve. Latência mais consistente que HDD
  • Premium SSD: produção crítica. IOPS alto garantido por tamanho de disco
  • Ultra Disk: banco de dados de alta exigência. Latência sub-milissegundo, IOPS configurável em runtime

SSE é criptografia da plataforma (padrão, transparente). ADE criptografa dentro do SO da VM (BitLocker/dm-crypt). Questão "criptografar o SO da VM com BitLocker" → ADE, não só SSE.

O que é: replica VMs inteiras para uma região secundária, permitindo failover em caso de desastre. É o serviço de Disaster Recovery — diferente de Backup.

  • RPO (Recovery Point Objective): quanto de dado você aceita perder. ASR atinge ~30 segundos
  • RTO (Recovery Time Objective): em quanto tempo você precisa estar online de novo. Depende do plano de recuperação
  • Replication: contínua e assíncrona da região primária para a secundária
  • Test failover: testa o DR sem impactar produção — usa rede isolada
  • Failover: ativa a região secundária de verdade
  • Reprotect: após failover, inverte a replicação (secundária vira origem)
  • Failback: volta a produção para a região primária
RPO é "quantas páginas do diário posso perder" (o quão recente é o último backup). RTO é "quanto tempo até eu reabrir a loja". Site Recovery mantém uma cópia viva pronta para assumir em outra cidade.
RPO = perda de dados aceitável (ASR ~30s). RTO = tempo para voltar. Backup restaura pontos no tempo; Site Recovery faz failover de VM inteira. Questão "failover de aplicação para outra região com perda mínima de dados" → Azure Site Recovery.

Role-Based Access Control. Controla quem pode fazer o quê, em qual nível. Você atribui uma role (papel), não permissões avulsas.

Um role assignment tem três partes:

  • O quê (Role Definition): conjunto de permissões
  • Onde (Scope): MG, Subscription, RG ou Resource
RBAC é um crachá: quem você é (principal), seu cargo (role) e o andar do prédio (scope).
2

  • Contributor: faz tudo exceto gerenciar acesso. Cria, modifica e deleta recursos
  • Reader: só visualiza
  • User Access Administrator: só gerencia IAM, não cria recursos. Oposto do Contributor
Contributor NÃO pode fazer role assignments. Para delegar acesso: Owner ou User Access Administrator.

A herança sempre desce, nunca sobe. Role atribuído em nível superior é herdado por todos os níveis inferiores.

MG → todas as Subs, RGs e Resources abaixo
Subscription → todos os RGs e Resources dentro
Resource Group → todos os Resources dentro
Resource → só esse recurso

Role assignments são aditivos — Reader na Sub + Contributor no RG X = Contributor no RG X, Reader no resto.

Boa prática: conceder permissão no nível mais alto necessário. Se todo o time precisa acessar tudo do projeto, atribua no RG, não em cada recurso individualmente.
Contributor na Sub A NÃO dá acesso à Sub B, mesmo no mesmo MG. Herança desce na árvore, nunca atravessa para outro galho.

Quando nenhuma role built-in atende exatamente, cria-se uma custom role.

  • Definida em JSON: Actions, NotActions, DataActions, AssignableScopes
  • Limite de 5000 custom roles por tenant
  • Criada no escopo de MG ou Subscription
  • Exemplo: pode reiniciar VMs mas não criar/deletar
NotActions não é negação explícita — é exclusão das Actions. Se outra role concede a ação, ela funciona. Para negar explicitamente → Deny Assignment ou Azure Policy Deny.
5
Deny Assignment — a negação explícitaDeny Assignment — the explicit denialnovo▾

O que é: uma atribuição que nega explicitamente ações sobre recursos, mesmo que o usuário tenha permissão via RBAC. É a única forma de "sobrescrever" um Contributor ou Owner com uma negação.

  • Prioridade: Deny Assignment sempre vence — se existe uma negação, a ação é bloqueada, não importa quantas roles a pessoa tenha
  • Criadas pelo próprio Azure em cenários específicos (ex: Azure Managed Applications, Blueprints deployments)
  • Você não cria Deny Assignments manualmente pelo portal como faz com role assignments
  • Aparecem em IAM → "Deny assignments"
  • Servem para proteger recursos gerenciados por serviços (ex: recursos criados por um Managed App não podem ser alterados pelo usuário final)
Deny Assignment é um lacre da polícia federal na porta. Não importa que você seja o dono do prédio (Owner) — se o lacre está ali, ninguém entra até ele ser removido pela autoridade que aplicou.
Ordem de avaliação: Deny Assignment > Role Assignment. Se você tem Owner mas existe um Deny Assignment sobre a ação, você é bloqueado. Diferença para Azure Policy: Deny Assignment nega ação de identidade; Policy nega tipo de recurso/configuração.

Policy effects — o que acontece quando violaPolicy effects — what happens on violationdecorarmemorize▾
  • Deny: bloqueia criação/modificação. Deploy falha. Uso: "VMs só em Brazil South"
  • Audit: permite, registra no Activity Log. Não bloqueia
  • AuditIfNotExists: audita se recurso relacionado não existe
  • Append: adiciona campo/tag na criação
  • Modify: adiciona, atualiza ou remove propriedades — permite remediation
  • DeployIfNotExists: deploy automático de recurso relacionado — permite remediation
  • Disabled: policy existe mas não avalia
Deny bloqueia criação. Audit só registra. Para compliance real → Deny. Para diagnóstico sem impacto → Audit.
3
Initiatives (Policy Sets) — agrupar policiesInitiatives (Policy Sets) — grouping policiesnovonew▾

O que é: uma Initiative (também chamada de Policy Set) é um conjunto de Azure Policies agrupadas para aplicar um mesmo padrão de governança. Em vez de atribuir 40 policies individualmente, você atribui a initiative de uma vez.

Objetivo: gerenciar diversas policies como uma única unidade — atribuir, monitorar compliance e reportar tudo junto.

  • Exemplo típico: uma initiative que combina "exigir tags obrigatórias", "exigir criptografia habilitada em storage" e "restrição de regiões" — tudo aplicado como um bloco
  • Built-in prontas: PCI-DSS, ISO 27001, NIST SP 800-53, CIS Azure Foundations Benchmark
  • Você pode criar initiatives customizadas misturando policies suas e built-in
  • Parâmetros da initiative são passados para as policies filhas — reuso e customização
  • Relatório de compliance mostra a initiative inteira e cada policy dentro individualmente
Initiative é um pacote de conformidade. Em vez de aplicar 40 regras PCI-DSS uma a uma (e correr o risco de esquecer), você aplica o pacote PCI-DSS de uma vez.
Questão típica: "aplicar simultaneamente exigência de tags, criptografia e restrição de regiões" → criar/atribuir uma Initiative, não policies avulsas.
4
Remediation tasks — corrigir o que já existeRemediation tasks — fixing what already existsnovonew▾

Policy avalia na criação. Recursos existentes antes da policy podem estar não conformes — remediation task corrige.

  • Só funciona com effects DeployIfNotExists ou Modify
  • Deny e Audit não têm remediation — Deny bloqueia futuros, Audit só registra
  • Requer managed identity na policy para executar a ação
  • Pode ser iniciada manualmente ou automaticamente após atribuição
Audit não tem remediation task. Questão "corrigir recursos existentes não conformes" → effect precisa ser DeployIfNotExists ou Modify.

⚠ Resumo — Policy & Initiative⚠ Summary — Policy & Initiative

Policy = conformidade (o que existe). RBAC = acesso (quem faz). Initiative = agrupar policies com objetivo comum. Deny bloqueia, Audit só registra. Remediation só com DeployIfNotExists ou Modify. Uma identidade com RBAC pode ser barrada por Policy.

Tags & Resource Locks

Organização, alocação de custo e proteção contra deleçãoOrganization, cost allocation and protection against deletion

ConceitosConcepts
1
Tags — metadados para organização e custoTags — metadata for organization and costnovonew▾

Par chave-valor aplicado em recursos e RGs para categorizar, filtrar e alocar custos. Exemplos: Environment=Production, CostCenter=Finance, Owner=ulysses@empresa.com.

  • Máximo de 50 tags por recurso (chave até 512 chars, valor até 256 chars)
  • Tags no RG não herdam automaticamente para recursos dentro
  • Case-insensitive nas chaves, mas Azure preserva o case original
  • Policy com effect Modify pode forçar tags em todos os recursos (com remediation)
  • Essenciais para FinOps — filtrar custo por projeto, equipe, ambiente
Tags no RG NÃO propagam para os recursos dentro. Para forçar tags em todos os recursos → Azure Policy com effect "Modify" + remediation task.
2
Azure Lock (Resource Locks) — proteção contra deleçãoAzure Lock (Resource Locks) — protection against deletionnovo▾

O que é: um lock impede modificação ou deleção de um recurso, independente das permissões RBAC. Pode ser aplicado em Resource, Resource Group ou Subscription. Mesmo um Owner é bloqueado — precisa remover o lock antes.

  • ReadOnly lock: nenhuma modificação ou deleção — equivale a forçar Reader para todos no recurso
  • Delete lock (CanNotDelete): pode modificar livremente, mas não pode deletar
  • Escopo: Resource, Resource Group ou Subscription — sempre herda para baixo
  • Lock no RG aplica para todos os recursos dentro
  • Lock na Subscription aplica para todos os RGs e recursos
  • Para remover: precisa de permissão Microsoft.Authorization/locks/delete
Lock é um cadeado extra na porta. O dono da casa (Owner) tem a chave-mestra, mas o cadeado precisa ser removido primeiro com uma chave separada — nem a chave-mestra o abre.
ReadOnly lock em Storage Account bloqueia listar chaves de acesso — operação que parece leitura mas internamente requer escrita. Pega candidatos em troubleshooting.

⚠ Resumo — Tags & Locks⚠ Summary — Tags & Locks

Tags no RG não herdam automaticamente (use Policy Modify). Locks funcionam independente do RBAC — nem Owner deleta com Delete lock. ReadOnly bloqueia operações que parecem leitura mas fazem escrita interna (listar keys do Storage).

Descoberta, classificação e proteção de dados em ambientes híbridos e multicloudDiscovery, classification and protection of data across hybrid and multicloud environments

ConceitosConcepts
1
Microsoft Purview — o que é e para que serveMicrosoft Purview — what it is and what it doesnovonew▾

O que é: solução unificada de governança, catalogação, classificação e proteção de dados da Microsoft. Enquanto Azure Policy governa recursos Azure, Purview governa os dados dentro (e fora) desses recursos.

Funções principais:

  • Descobrir: escaneia fontes de dados e identifica automaticamente onde estão as informações
  • Classificar: aplica rótulos como "Confidencial", "PII", "Financeiro" — automaticamente ou manualmente
  • Catalogar: mantém um catálogo pesquisável de todos os dados da organização (data catalog)
  • Monitorar: acompanha lineage (de onde o dado veio e para onde foi), acesso e compliance

Escopo de fontes: Azure (Storage, SQL, Data Lake), Microsoft 365, ambientes on-premises e outras nuvens (AWS S3, Google Cloud) — é multicloud por design.

  • Objetivo: apoiar governança de dados, conformidade regulatória (LGPD, GDPR) e proteção de informações sensíveis
  • Ajuda a responder perguntas como: "onde estão meus dados de PII?", "quem acessou este dataset?", "quais tabelas alimentam este relatório?"
Se Azure Policy é o código de obras (o que pode ser construído), Purview é o cartório de imóveis: sabe onde cada coisa está, quem é o dono, o histórico de vendas e as restrições legais de cada terreno.
Não confunda: Azure Policy governa recursos (VMs, storage). Purview governa os dados dentro deles. Uma organização usa os dois — são complementares. Para o dia a dia de administração Azure, saber Purview em nível conceitual (o que é, para que serve, escopo) já cobre bem — o aprofundamento fica para especialistas de dados.

Pegadinhas clássicasClassic traps

Erros comuns até de quem já tem experiência. Leia cada uma com atenção.Common mistakes even from experienced professionals. Read each one carefully.

⚠ Tenant ≠ Subscription

Tenant é o diretório do Entra ID (identidades). Subscription é o contrato de billing (recursos). Um tenant tem várias subs, cada sub pertence a um tenant. Ter conta no Entra ID não dá acesso a nenhuma sub — precisa de RBAC.

⚠ Entra ID ≠ Active Directory Domain Services

Entra ID (nuvem, web) usa OAuth/SAML/OpenID. AD DS (on-prem) usa LDAP/Kerberos. Não são o mesmo serviço. Questão mencionando "Azure AD" está falando de Entra ID — o nome mudou em 2023.

⚠ NSG vs Azure Firewall

NSG = sem FQDN filtering, logging limitado. Azure Firewall = FQDN, logging completo, IDPS. Questão com "filtrar por domínio" → Azure Firewall.

⚠ Load Balancer vs Application Gateway

LB é L4 — sem HTTP, sem WAF, sem path routing. App Gateway é L7 — path-based routing, WAF, cookie affinity. Questão com "path-based routing" ou "WAF" → App Gateway.

⚠ RBAC scope — herança é só para baixo

Role na Sub A NÃO afeta Sub B, mesmo no mesmo MG. Herança: MG → Sub → RG → Resource. Nunca sobe, nunca atravessa.

⚠ Deny Assignment vence RBAC

Se existe Deny Assignment sobre uma ação, mesmo Owner é bloqueado. Você não cria manualmente — são criadas pelo Azure em Managed Apps/Blueprints. NotActions em custom role NÃO é o mesmo — não é negação real.

⚠ Contributor não delega acesso

Contributor faz tudo menos IAM. Para delegar acesso a outro usuário → Owner ou User Access Administrator.

⚠ Service Endpoint vs Private Endpoint

SE roteia pelo backbone Azure, mas PaaS ainda tem IP público. PE cria NIC com IP privado — serviço 100% privado. Compliance estrito → Private Endpoint.

⚠ Archive tier — rehydration obrigatória

Blob em Archive NÃO pode ser lido diretamente. Rehydration: até 15h Standard ou ~1h High Priority. "Acesso imediato" → Archive está errado.

⚠ Managed Identity System vs User-assigned

System: 1:1 com recurso, não compartilhável. User: ciclo independente, compartilhável entre múltiplos recursos. "Mesma identidade em várias VMs" → User-assigned.

⚠ VNet Peering — não é transitivo

A↔B e B↔C não significa A↔C. Hub-and-spoke com Spokes se comunicando → UDR + NVA/Firewall no Hub.

⚠ Availability Set ≠ Availability Zone

AS distribui VMs dentro do mesmo datacenter (SLA 99.95%). AZ separa em datacenters distintos (SLA 99.99%). "Sobreviver à perda de um datacenter" → Availability Zones.

⚠ Dynamic Group exige P1; PIM exige P2

Conditional Access e Dynamic Groups → P1. PIM (just-in-time roles) e Identity Protection → P2. Confundir licenças é erro comum.

⚠ Tags no RG não herdam automaticamente

Tags no RG NÃO propagam para recursos dentro. Para forçar em todos → Azure Policy com effect "Modify" + remediation task.

⚠ Budget não bloqueia gasto

Budget só alerta ao atingir o limite — não interrompe recursos. "Garantir que gasto não ultrapasse X" → budget + Azure Policy restringindo criação de recursos caros.

⚠ GRS vs RA-GRS

GRS: secundário só após failover. RA-GRS: leitura do secundário sempre disponível (sufixo -secondary). "Leitura da secundária sem failover" → RA-GRS.

⚠ Cloud Shell precisa de Storage Account

Sem Azure File Share configurado, Cloud Shell roda em modo efêmero — arquivos somem ao fechar. Persistência = File Share associado.

⚠ Advisor recomenda, não aplica

Advisor sugere melhorias em 5 categorias mas não aplica sozinho. Você precisa agir sobre cada recomendação. Não confunda com Defender for Cloud (foco em segurança).
