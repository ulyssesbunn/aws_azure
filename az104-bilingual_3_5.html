Tenant — o que é e por que é a base de tudoTenant — what it is and why it is the foundation of everythingfundaçãofoundation▾

O que é: um tenant é uma instância dedicada e isolada do Microsoft Entra ID que sua organização recebe ao assinar um serviço Microsoft (Azure, Microsoft 365). É o "diretório" onde vivem todas as suas identidades — usuários, grupos, apps.

What it is: a tenant is a dedicated, isolated instance of Microsoft Entra ID that your organization receives when subscribing to a Microsoft service (Azure, Microsoft 365). It is the "directory" where all your identities live — users, groups, apps.

  • Cada tenant tem um domínio inicial tipo empresa.onmicrosoft.com (você pode adicionar domínios customizados depois)
  • Tem um Tenant ID — um GUID único que identifica seu diretório
  • É um limite de segurança e identidade: usuários de um tenant não existem em outro por padrão
  • Uma organização normalmente tem um tenant, mas pode ter vários (ex: fusões, separação prod/test)
  • Toda subscription do Azure é associada a exatamente um tenant para autenticação
  • Each tenant has an initial domain like company.onmicrosoft.com (you can add custom domains later)
  • Has a Tenant ID — a unique GUID that identifies your directory
  • It is a security and identity boundary: users from one tenant do not exist in another by default
  • An organization typically has one tenant, but can have several (e.g., mergers, prod/test separation)
  • Every Azure subscription is associated with exactly one tenant for authentication
O tenant é o prédio da sua empresa com sua própria portaria e lista de funcionários. Ninguém de outro prédio entra sem ser convidado (Guest). As subscriptions são os contratos de serviço que esse prédio assina.
The tenant is your company's building with its own front desk and employee list. Nobody from another building enters without being invited (Guest). Subscriptions are the service contracts that building signs.
Tenant ≠ Subscription. Tenant é o diretório de identidades (Entra ID). Subscription é o contrato de billing dos serviços. Um tenant pode ter várias subscriptions, mas cada subscription pertence a um único tenant.
Tenant ≠ Subscription. Tenant is the identity directory (Entra ID). Subscription is the billing contract for services. One tenant can have several subscriptions, but each subscription belongs to a single tenant.

Microsoft Entra ID — o serviço de identidade (ex-Azure AD)Microsoft Entra ID — the identity service (formerly Azure AD)fundaçãofoundation▾

O que é: o serviço de gerenciamento de identidade e acesso da Microsoft na nuvem. Era chamado de Azure Active Directory (Azure AD) até 2023 — o nome mudou para Microsoft Entra ID, mas a função é a mesma. É onde você gerencia quem pode acessar o quê.

What it is: Microsoft's cloud-based identity and access management service. It was called Azure Active Directory (Azure AD) until 2023 — the name changed to Microsoft Entra ID, but the function is the same. This is where you manage who can access what.

  • Autenticação (AuthN): confirma que você é quem diz ser — login, MFA
  • Autorização (AuthZ): define o que você pode fazer — trabalha junto com RBAC
  • Gerencia usuários, grupos, dispositivos, aplicações e service principals
  • Base para SSO (Single Sign-On) entre Azure, Microsoft 365 e apps terceiros
  • Suporta identidades híbridas — sincroniza com AD on-premises via Entra Connect
  • Não é o mesmo que Active Directory (AD DS) on-premises — Entra ID é baseado em nuvem, usa protocolos web (OAuth, SAML, OpenID Connect), não LDAP/Kerberos
  • Authentication (AuthN): confirms you are who you say you are — sign-in, MFA
  • Authorization (AuthZ): defines what you can do — works together with RBAC
  • Manages users, groups, devices, applications, and service principals
  • Foundation for SSO (Single Sign-On) across Azure, Microsoft 365, and third-party apps
  • Supports hybrid identities — syncs with on-premises AD via Entra Connect
  • Not the same as Active Directory (AD DS) on-premises — Entra ID is cloud-based, uses web protocols (OAuth, SAML, OpenID Connect), not LDAP/Kerberos
Entra ID é a portaria digital da empresa. Ele verifica o crachá (autenticação), consulta a lista do que você pode acessar (autorização) e libera as portas certas — tudo pela nuvem, não por servidor físico.
Entra ID is the company's digital front desk. It checks your badge (authentication), consults the list of what you can access (authorization), and unlocks the right doors — all through the cloud, not on a physical server.
Entra ID (nuvem) ≠ Active Directory Domain Services (on-premises). AD DS usa LDAP/Kerberos e roda em servidores da empresa. Entra ID usa protocolos web e roda na nuvem Microsoft. Questão que mencionar "Azure AD" está falando de Entra ID — é o mesmo serviço, nome antigo.
Entra ID (cloud) ≠ Active Directory Domain Services (on-premises). AD DS uses LDAP/Kerberos and runs on company servers. Entra ID uses web protocols and runs on the Microsoft cloud. Anything mentioning "Azure AD" is referring to Entra ID — same service, old name.

Users & Groups — tipos e diferençasUsers & Groups — types and differencesdecorarmemorize▾

Toda identidade humana no Entra ID é um user, e usuários são organizados em groups para facilitar a atribuição de acesso em massa.

Every human identity in Entra ID is a user, and users are organized into groups to simplify access assignment at scale.

  • Member user: conta interna da organização (funcionário)
  • Guest user (B2B): conta externa convidada — parceiro, fornecedor; usa a identidade da própria empresa dele
  • Security group: usado para atribuir RBAC, Policy e acesso a recursos
  • Microsoft 365 group: usado para colaboração (email compartilhado, Teams, SharePoint)
  • Assigned membership: você adiciona/remove membros manualmente
  • Dynamic membership: membros entram/saem automaticamente por regra de atributo — exige licença P1
  • UPN (User Principal Name): o login, tipo ana@empresa.com — pode mudar
  • Object ID: GUID imutável do usuário — use em scripts, nunca muda
  • Member user: internal organization account (employee)
  • Guest user (B2B): invited external account — partner, vendor; uses their own company's identity
  • Security group: used to assign RBAC, Policy, and resource access
  • Microsoft 365 group: used for collaboration (shared inbox, Teams, SharePoint)
  • Assigned membership: you add/remove members manually
  • Dynamic membership: members join/leave automatically by attribute rule — requires P1 license
  • UPN (User Principal Name): the sign-in name, like ana@company.com — can change
  • Object ID: immutable GUID of the user — use in scripts, never changes
Atribuir acesso a grupos em vez de usuários individuais é como dar chaves a cargos, não a pessoas. Quando alguém troca de função, você só muda o grupo — não precisa refazer 20 permissões.
Assigning access to groups instead of individual users is like giving keys to roles, not to people. When someone changes jobs, you just change the group — no need to redo 20 permissions.
Dynamic Groups exigem Entra ID P1. Você NÃO pode adicionar membros manualmente em um dynamic group — eles entram por regra de atributo. Questão "automaticamente adicionar novos funcionários do departamento X" → Dynamic Security Group + P1.
Dynamic Groups require Entra ID P1. You CANNOT manually add members to a dynamic group — they join by attribute rule. Question about "automatically add new employees from department X" → Dynamic Security Group + P1.

Licenças Entra ID — Free, P1 e P2Entra ID Licenses — Free, P1 and P2decorarmemorize▾

Muitas features importantes só existem em certas licenças. Saber qual feature precisa de qual licença é decisivo.

Many important features only exist in certain licenses. Knowing which feature requires which license is decisive.

  • Free: usuários, grupos, SSO básico, MFA de segurança básica
  • P1: Conditional Access, Dynamic Groups, self-service password reset com writeback, gerenciamento avançado de grupos
  • P2: tudo do P1 + PIM (Privileged Identity Management), Identity Protection (detecção de risco), access reviews
  • Free: users, groups, basic SSO, basic security MFA
  • P1: Conditional Access, Dynamic Groups, self-service password reset with writeback, advanced group management
  • P2: everything in P1 + PIM (Privileged Identity Management), Identity Protection (risk detection), access reviews
As licenças são planos de assinatura crescentes. Free é o básico que todo mundo tem, P1 adiciona controle de acesso condicional, P2 adiciona segurança avançada e acesso just-in-time (PIM).
Licenses are progressive subscription tiers. Free is the baseline everyone gets, P1 adds conditional access control, P2 adds advanced security and just-in-time access (PIM).
Conditional Access e Dynamic Groups → P1. PIM e Identity Protection → P2. Se a questão pede acesso "just-in-time" a roles privilegiadas → PIM → licença P2.
Conditional Access and Dynamic Groups → P1. PIM and Identity Protection → P2. If the question asks for "just-in-time" access to privileged roles → PIM → P2 license.

MFA & Conditional Access — segurança de acessoMFA & Conditional Access — access securitynovonew▾

MFA (Multi-Factor Authentication): exige um segundo fator além da senha (app autenticador, SMS, token). Conditional Access: políticas "se-então" que decidem quando exigir MFA ou bloquear acesso.

MFA (Multi-Factor Authentication): requires a second factor beyond the password (authenticator app, SMS, token). Conditional Access: "if-then" policies that decide when to require MFA or block access.

  • Conditional Access avalia sinais: usuário, localização, dispositivo, app, risco de login
  • Exemplo de política: "se login vier de fora do Brasil, exigir MFA"
  • Exemplo: "se dispositivo não estiver conforme (não gerenciado), bloquear acesso"
  • Conditional Access exige licença P1
  • Security Defaults: MFA básico gratuito para todos, mas sem granularidade
  • Conditional Access evaluates signals: user, location, device, app, sign-in risk
  • Example policy: "if sign-in comes from outside Brazil, require MFA"
  • Example: "if device is not compliant (unmanaged), block access"
  • Conditional Access requires P1 license
  • Security Defaults: basic MFA free for everyone, but no granularity
Conditional Access é o segurança inteligente da portaria: ele não pede documento de todo mundo igual — pede identidade extra (MFA) só de quem chega em horário estranho ou de local desconhecido.
Conditional Access is the intelligent security guard at the door: it doesn't ask everyone for ID the same way — it asks for extra verification (MFA) only from those arriving at odd hours or from unknown locations.
Security Defaults (grátis) é tudo-ou-nada. Conditional Access (P1) permite regras granulares. Questão que pede "exigir MFA só para admins" ou "só de fora da rede corporativa" → Conditional Access → P1.
Security Defaults (free) is all-or-nothing. Conditional Access (P1) allows granular rules. Any question asking "require MFA only for admins" or "only from outside the corporate network" → Conditional Access → P1.

Service Principal & Managed Identity — identidade para appsService Principal & Managed Identity — identity for appsmais cobradomost tested▾

Nem toda identidade é humana. Apps, scripts e serviços também precisam se autenticar — e é aí que entram Service Principals e Managed Identities.

Not every identity is human. Apps, scripts, and services also need to authenticate — that's where Service Principals and Managed Identities come in.

  • Service Principal: a identidade de uma aplicação dentro do tenant. Tem credencial (client secret ou certificado) que você precisa gerenciar e rotacionar
  • Managed Identity System-assigned: criada e vinculada a UM recurso (ex: uma VM). Deletada junto com o recurso. Não pode ser compartilhada
  • Managed Identity User-assigned: criada como recurso independente, pode ser atribuída a vários recursos ao mesmo tempo
  • A grande vantagem da Managed Identity: o Azure gerencia a credencial — sem secret no código, sem rotação manual
  • Managed Identity é, por baixo, um tipo especial de Service Principal gerenciado pela plataforma
  • Service Principal: the identity of an application inside the tenant. Has a credential (client secret or certificate) that you must manage and rotate
  • System-assigned Managed Identity: created and tied to ONE resource (e.g., a VM). Deleted with the resource. Cannot be shared
  • User-assigned Managed Identity: created as an independent resource, can be assigned to multiple resources at the same time
  • The big advantage of Managed Identity: Azure manages the credential — no secret in code, no manual rotation
  • Under the hood, Managed Identity is a special kind of Service Principal managed by the platform
Service Principal é um crachá de robô que você mesmo tem que renovar. Managed Identity é o mesmo crachá, mas o próprio prédio renova automaticamente — você nunca lida com a validade.
Service Principal is a robot badge that you have to renew yourself. Managed Identity is the same badge, but the building renews it automatically — you never deal with expiration.
System-assigned: 1:1 com o recurso, morre com ele. User-assigned: compartilhável entre vários recursos. Questão "múltiplas VMs precisam da mesma identidade para acessar o Key Vault" → User-assigned Managed Identity.
System-assigned: 1:1 with the resource, dies with it. User-assigned: shareable across multiple resources. Question "multiple VMs need the same identity to access Key Vault" → User-assigned Managed Identity.

Self-Service Password Reset (SSPR) — usuário resolve sozinhoSelf-Service Password Reset (SSPR) — users solve it themselvesAZ-104▾

O que é: funcionalidade que permite que usuários redefinam a própria senha sem precisar acionar o suporte técnico/help desk, através de métodos de verificação configurados previamente.

What it is: a feature that lets users reset their own password without contacting the help desk, using verification methods configured in advance.

  • Métodos de autenticação: telefone (SMS/chamada), email alternativo, Microsoft Authenticator app, security questions
  • Administrador define quantos métodos são obrigatórios para confirmar identidade (ex: 2 de 4)
  • Combined Registration: registro unificado de métodos de SSPR e MFA — o usuário configura uma vez, serve para os dois
  • Requer licença Free (limitado) ou P1 (com writeback para sincronizar a nova senha de volta ao AD DS on-prem)
  • Reduz drasticamente chamados de help desk relacionados a senha esquecida
  • Authentication methods: phone (SMS/call), alternate email, Microsoft Authenticator app, security questions
  • The admin defines how many methods are required to confirm identity (e.g., 2 of 4)
  • Combined Registration: unified registration of SSPR and MFA methods — the user sets it up once, it serves both
  • Requires Free (limited) or P1 licensing (with writeback to sync the new password back to on-prem AD DS)
  • Drastically reduces help-desk tickets related to forgotten passwords
SSPR "writeback" para sincronizar a senha de volta para o Active Directory on-prem exige licença P1 — a versão gratuita funciona apenas para contas nativas do Entra ID (cloud-only).
SSPR "writeback" to sync the password back to on-prem Active Directory requires a P1 license — the free tier only works for native Entra ID (cloud-only) accounts.

Administrative Units — delegação administrativa por área organizacionalAdministrative Units — administrative delegation by organizational areaAZ-104▾

O que é: recurso do Entra ID que agrupa usuários, grupos ou dispositivos (ex: por filial, departamento ou escola) para permitir delegação administrativa restrita a esse subconjunto, sem dar acesso a todo o tenant.

What it is: an Entra ID resource that groups users, groups, or devices (e.g., by branch, department, or school) to allow administrative delegation restricted to that subset, without granting access to the entire tenant.

  • Um administrador atribuído a uma Administrative Unit só gerencia os membros daquela unidade — não o tenant inteiro
  • Suporta roles como Helpdesk Administrator, User Administrator, Groups Administrator com escopo restrito à AU
  • Um usuário/grupo pode pertencer a múltiplas Administrative Units simultaneamente
  • Pode ter membership dinâmica baseada em regras de atributo, similar a Dynamic Groups
  • Caso de uso clássico: universidade com uma AU por faculdade, cada uma com seu próprio "admin local"
  • An admin assigned to an Administrative Unit only manages that unit's members — not the entire tenant
  • Supports roles like Helpdesk Administrator, User Administrator, Groups Administrator scoped to the AU
  • A user/group can belong to multiple Administrative Units at the same time
  • Can have dynamic membership based on attribute rules, similar to Dynamic Groups
  • Classic use case: a university with one AU per college, each with its own "local admin"
Administrative Units são como diretores de filial: cada um administra só a própria loja (contratando, demitindo, resetando senha dos funcionários locais), sem poder mexer em outra filial nem na matriz.
Administrative Units are like branch managers: each one administers only their own store (hiring, firing, resetting local employees' passwords), without touching another branch or headquarters.
Administrative Units delegam administração de IDENTIDADE (usuários/grupos/dispositivos) por área organizacional. Não confundir com Management Groups, que organizam SUBSCRIPTIONS para RBAC/Policy de recursos — são hierarquias completamente diferentes.
Administrative Units delegate IDENTITY administration (users/groups/devices) by organizational area. Don't confuse with Management Groups, which organize SUBSCRIPTIONS for resource RBAC/Policy — completely different hierarchies.

PIM avançado — Eligible vs Active, e acesso Just-In-TimeAdvanced PIM — Eligible vs Active, and Just-In-Time accessAZ-104▾

Privileged Identity Management (PIM) já aparece no material como "acesso just-in-time a roles privilegiadas" — aqui aprofundamos os dois estados centrais de uma atribuição PIM.

Privileged Identity Management (PIM) already appears in the material as "just-in-time access to privileged roles" — here we go deeper into the two central states of a PIM assignment.

  • Eligible (elegível): o usuário PODE ativar a role quando precisar, mas não a tem ativa por padrão — precisa fazer "Activate" para ganhar o privilégio por um período limitado
  • Active (ativa): o usuário JÁ possui a role em uso, igual a uma atribuição RBAC tradicional — sem precisar ativar nada
  • Just-In-Time (JIT): o princípio por trás de "Eligible" — o privilégio existe apenas pelo tempo necessário, reduzindo a superfície de ataque de contas sempre-privilegiadas
  • Ativação pode exigir MFA, justificativa por texto e aprovação de um segundo administrador
  • Toda ativação gera auditoria — quem ativou, quando, por quanto tempo e com qual justificativa
  • Assignments "Active" também podem ter data de expiração, mas não passam pelo fluxo de ativação
  • Eligible: the user CAN activate the role when needed, but doesn't have it active by default — must "Activate" to gain the privilege for a limited time
  • Active: the user ALREADY has the role in use, just like a traditional RBAC assignment — nothing to activate
  • Just-In-Time (JIT): the principle behind "Eligible" — the privilege exists only for as long as needed, reducing the attack surface of always-privileged accounts
  • Activation can require MFA, a text justification, and approval from a second administrator
  • Every activation generates an audit trail — who activated, when, for how long, and with what justification
  • "Active" assignments can also have an expiration date, but they don't go through the activation flow
Eligible é como ter a chave-mestra guardada no cofre da portaria: você precisa "pedir e assinar" (Activate) para usá-la por um tempo limitado. Active é andar com a chave-mestra no bolso o tempo todo — mais conveniente, porém mais arriscado se for roubada.
Eligible is like keeping the master key locked in the front-desk safe: you need to "request and sign" (Activate) to use it for a limited time. Active is carrying the master key in your pocket all the time — more convenient, but riskier if it's stolen.
Questão "reduzir o tempo que uma conta de administrador tem privilégio elevado" → PIM com atribuição Eligible (JIT), não Active. Active ainda é PIM, mas sem o benefício de "privilégio zero até precisar".
Question "reduce the time an admin account holds elevated privilege" → PIM with an Eligible assignment (JIT), not Active. Active is still PIM, but without the "zero privilege until needed" benefit.

NSG & ASG — controle de tráfegoNSG & ASG — traffic controlmais cobradomost tested▾

NSG (Network Security Group): firewall básico com regras de permitir/negar tráfego. ASG (Application Security Group): agrupa VMs logicamente para usar nas regras do NSG.

NSG (Network Security Group): basic firewall with allow/deny traffic rules. ASG (Application Security Group): logically groups VMs to use in NSG rules.

  • Stateful: se você permite tráfego de entrada, a resposta de saída é automática — não precisa regra de retorno
  • Priority: menor número = maior prioridade (100 vence 200). Range 100–4096
  • Regras default: AllowVNetInBound, AllowAzureLoadBalancerInBound, DenyAllInBound (menor prioridade)
  • Associação: NSG pode estar na subnet (afeta todas as VMs) ou na NIC (afeta uma VM)
  • Effective rules: se houver NSG na subnet E na NIC, ambos são avaliados — os dois precisam permitir
  • ASG: em vez de listar IPs nas regras, agrupa NICs — ex: regra "permitir do ASG-Web para o ASG-DB"
  • Stateful: if you allow inbound traffic, the outbound response is automatic — no return rule needed
  • Priority: lower number = higher priority (100 wins over 200). Range 100–4096
  • Default rules: AllowVNetInBound, AllowAzureLoadBalancerInBound, DenyAllInBound (lowest priority)
  • Association: NSG can be on the subnet (affects all VMs) or on the NIC (affects one VM)
  • Effective rules: if there is an NSG on the subnet AND on the NIC, both are evaluated — both must allow
  • ASG: instead of listing IPs in rules, groups NICs — e.g., rule "allow from ASG-Web to ASG-DB"
NSG é a lista de regras do porteiro (quem entra, quem sai). ASG é dar apelidos aos grupos — em vez de dizer "libere para 10.0.1.4, 10.0.1.5, 10.0.1.6", você diz "libere para o grupo Web".
NSG is the doorman's rulebook (who comes in, who goes out). ASG is giving nicknames to groups — instead of "allow for 10.0.1.4, 10.0.1.5, 10.0.1.6", you say "allow for the Web group".
NSG é stateful — não precisa de regra de retorno. Se houver NSG na subnet e na NIC, o tráfego precisa passar por AMBOS. Menor número de prioridade vence. Questão de troubleshooting: "tráfego bloqueado" geralmente é regra de prioridade menor negando antes.
NSG is stateful — no return rule needed. If there is an NSG on the subnet and on the NIC, traffic must pass through BOTH. Lower priority number wins. Troubleshooting: "blocked traffic" is usually a lower-priority rule denying first.

O que é: conecta duas VNets para que se comuniquem como se fossem uma só, usando o backbone privado do Azure (sem passar pela internet).

What it is: connects two VNets so they communicate as if they were one, using Azure's private backbone (without going through the internet).

  • Regional peering: VNets na mesma região
  • Global peering: VNets em regiões diferentes — maior latência, cobra egress
  • NÃO é transitivo: se A↔B e B↔C, isso NÃO significa A↔C
  • Para tráfego transitivo (hub-and-spoke): precisa de UDR + Azure Firewall/NVA no hub
  • Peering exige address spaces sem sobreposição
  • Regional peering: VNets in the same region
  • Global peering: VNets in different regions — higher latency, charges egress
  • NOT transitive: if A↔B and B↔C, that does NOT mean A↔C
  • For transitive traffic (hub-and-spoke): requires UDR + Azure Firewall/NVA in the hub
  • Peering requires non-overlapping address spaces
Peering é construir uma ponte direta entre dois terrenos. Só porque o terreno A tem ponte para o B, e o B tem ponte para o C, não existe uma ponte automática de A para C — você teria que construí-la ou passar por um pedágio central (firewall no hub).
Peering is building a direct bridge between two lots. Just because lot A has a bridge to B, and B has a bridge to C, does not create an automatic bridge from A to C — you would have to build it or go through a central toll (firewall in the hub).
VNet Peering NÃO é transitivo. Em hub-and-spoke, Spoke1 não fala com Spoke2 automaticamente. Solução: UDR (User Defined Route) apontando para Azure Firewall/NVA no Hub.

Escolher o balanceador certo é uma das decisões mais cobradas. A chave é: camada (L4 vs L7), escopo (regional vs global) e se precisa de WAF.

Choosing the right load balancer is one of the most tested decisions. The key: layer (L4 vs L7), scope (regional vs global), and whether WAF is needed.

  • Azure Load Balancer: L4 (TCP/UDP), regional. Não entende HTTP, sem WAF. Para VMs e VMSS
  • Application Gateway: L7 (HTTP/HTTPS), regional. WAF, path-based routing, cookie affinity, TLS termination
  • Traffic Manager: baseado em DNS, global. Roteamento por latência/prioridade/peso. Não inspeciona conteúdo
  • Azure Front Door: L7 global, Anycast, CDN integrado, WAF global, failover automático
  • Regra de bolso: HTTP global → Front Door | HTTP regional → App Gateway | TCP/UDP → Load Balancer | DNS global → Traffic Manager
  • Azure Load Balancer: L4 (TCP/UDP), regional. Does not understand HTTP, no WAF. For VMs and VMSS
  • Application Gateway: L7 (HTTP/HTTPS), regional. WAF, path-based routing, cookie affinity, TLS termination
  • Traffic Manager: DNS-based, global. Routing by latency/priority/weight. Does not inspect content
  • Azure Front Door: Global L7, Anycast, integrated CDN, global WAF, automatic failover
  • Rule of thumb: Global HTTP → Front Door | Regional HTTP → App Gateway | TCP/UDP → Load Balancer | Global DNS → Traffic Manager

Load Balancer is the doorman who directs people into queues without looking at the content. App Gateway is the receptionist who reads the request ("need support? go to counter 3") and even blocks suspicious visitors (WAF).

Para o dia a dia de administração Azure, saber Purview em nível conceitual (o que é, para que serve, escopo) já cobre bem — o aprofundamento fica para especialistas de dados.
Do not confuse: Azure Policy governs resources (VMs, storage). Purview governs the data inside them. An organization uses both — they are complementary. For day-to-day Azure administration, knowing Purview at the conceptual level (what it is, what it does, scope) is enough — deep dives are for data specialists.

Pegadinhas clássicasClassic traps

Erros comuns até de quem já tem experiência. Leia cada uma com atenção.Common mistakes even from experienced professionals. Read each one carefully.

⚠ Tenant ≠ Subscription

Tenant é o diretório do Entra ID (identidades). Subscription é o contrato de billing (recursos). Um tenant tem várias subs, cada sub pertence a um tenant. Ter conta no Entra ID não dá acesso a nenhuma sub — precisa de RBAC.

Tenant is the Entra ID directory (identities). Subscription is the billing contract (resources). One tenant has many subs, each sub belongs to one tenant. Having an Entra ID account does not grant access to any sub — RBAC is required.

⚠ Entra ID ≠ Active Directory Domain Services

Entra ID (nuvem, web) usa OAuth/SAML/OpenID. AD DS (on-prem) usa LDAP/Kerberos. Não são o mesmo serviço. Questão mencionando "Azure AD" está falando de Entra ID — o nome mudou em 2023.

Entra ID (cloud, web) uses OAuth/SAML/OpenID. AD DS (on-prem) uses LDAP/Kerberos. Not the same service. Anything mentioning "Azure AD" is referring to Entra ID — renamed in 2023.

⚠ NSG vs Azure Firewall

NSG = sem FQDN filtering, logging limitado. Azure Firewall = FQDN, logging completo, IDPS. Questão com "filtrar por domínio" → Azure Firewall.

NSG = no FQDN filtering, limited logging. Azure Firewall = FQDN, full logging, IDPS. Question with "filter by domain" → Azure Firewall.

⚠ Load Balancer vs Application Gateway

LB é L4 — sem HTTP, sem WAF, sem path routing. App Gateway é L7 — path-based routing, WAF, cookie affinity. Questão com "path-based routing" ou "WAF" → App Gateway.

LB is L4 — no HTTP, no WAF, no path routing. App Gateway is L7 — path-based routing, WAF, cookie affinity. Question with "path-based routing" or "WAF" → App Gateway.

⚠ RBAC scope — herança é só para baixo⚠ RBAC scope — inheritance only flows down

Role na Sub A NÃO afeta Sub B, mesmo no mesmo MG. Herança: MG → Sub → RG → Resource. Nunca sobe, nunca atravessa.

A role on Sub A does NOT affect Sub B, even under the same MG. Inheritance: MG → Sub → RG → Resource. Never up, never sideways.

⚠ Deny Assignment vence RBAC⚠ Deny Assignment beats RBAC

Se existe Deny Assignment sobre uma ação, mesmo Owner é bloqueado. Você não cria manualmente — são criadas pelo Azure em Managed Apps/Blueprints. NotActions em custom role NÃO é o mesmo — não é negação real.

If a Deny Assignment covers an action, even Owner is blocked. You don't create it manually — Azure creates them via Managed Apps/Blueprints. NotActions in a custom role is NOT the same — it's not real denial.

⚠ Contributor não delega acesso⚠ Contributor cannot delegate access

Contributor faz tudo menos IAM. Para delegar acesso a outro usuário → Owner ou User Access Administrator.

Contributor does everything except IAM. To delegate access to another user → Owner or User Access Administrator.

⚠ Service Endpoint vs Private Endpoint

SE roteia pelo backbone Azure, mas PaaS ainda tem IP público. PE cria NIC com IP privado — serviço 100% privado. Compliance estrito → Private Endpoint.

SE routes through the Azure backbone, but PaaS still has a public IP. PE creates a NIC with a private IP — 100% private service. Strict compliance → Private Endpoint.

⚠ Archive tier — rehydration obrigatória⚠ Archive tier — rehydration required

Blob em Archive NÃO pode ser lido diretamente. Rehydration: até 15h Standard ou ~1h High Priority. "Acesso imediato" → Archive está errado.

Blob in Archive CANNOT be read directly. Rehydration: up to 15h Standard or ~1h High Priority. "Immediate access" → Archive is wrong.

⚠ Managed Identity System vs User-assigned

System: 1:1 com recurso, não compartilhável. User: ciclo independente, compartilhável entre múltiplos recursos. "Mesma identidade em várias VMs" → User-assigned.

System: 1:1 with resource, not shareable. User: independent lifecycle, shareable across multiple resources. "Same identity on multiple VMs" → User-assigned.

⚠ VNet Peering — não é transitivo⚠ VNet Peering — not transitive

A↔B e B↔C não significa A↔C. Hub-and-spoke com Spokes se comunicando → UDR + NVA/Firewall no Hub.

A↔B and B↔C does not mean A↔C. Hub-and-spoke with Spokes talking to each other → UDR + NVA/Firewall in the Hub.

⚠ Availability Set ≠ Availability Zone

AS distribui VMs dentro do mesmo datacenter (SLA 99.95%). AZ separa em datacenters distintos (SLA 99.99%). "Sobreviver à perda de um datacenter" → Availability Zones.

AS distributes VMs within the same datacenter (SLA 99.95%). AZ separates them across distinct datacenters (SLA 99.99%). "Survive a datacenter loss" → Availability Zones.

⚠ Dynamic Group exige P1; PIM exige P2⚠ Dynamic Group requires P1; PIM requires P2

Conditional Access e Dynamic Groups → P1. PIM (just-in-time roles) e Identity Protection → P2. Confundir licenças é erro comum.

Conditional Access and Dynamic Groups → P1. PIM (just-in-time roles) and Identity Protection → P2. Mixing up licenses is a common mistake.

⚠ Tags no RG não herdam automaticamente⚠ Tags on the RG don't inherit automatically

Tags no RG NÃO propagam para recursos dentro. Para forçar em todos → Azure Policy com effect "Modify" + remediation task.

Tags on the RG do NOT propagate to resources inside. To enforce on all → Azure Policy with "Modify" effect + remediation task.

⚠ Budget não bloqueia gasto⚠ Budget does not block spending

Budget só alerta ao atingir o limite — não interrompe recursos. "Garantir que gasto não ultrapasse X" → budget + Azure Policy restringindo criação de recursos caros.

Budget only alerts when the limit is reached — it does not stop resources. "Ensure spending stays under X" → budget + Azure Policy restricting creation of expensive resources.

⚠ GRS vs RA-GRS

GRS: secundário só após failover. RA-GRS: leitura do secundário sempre disponível (sufixo -secondary). "Leitura da secundária sem failover" → RA-GRS.

GRS: secondary only after failover. RA-GRS: secondary read always available (-secondary suffix). "Read from secondary without failover" → RA-GRS.

⚠ Cloud Shell precisa de Storage Account⚠ Cloud Shell needs a Storage Account

Sem Azure File Share configurado, Cloud Shell roda em modo efêmero — arquivos somem ao fechar. Persistência = File Share associado.

Without an Azure File Share configured, Cloud Shell runs in ephemeral mode — files disappear on close. Persistence = associated File Share.

⚠ Advisor recomenda, não aplica⚠ Advisor recommends, does not apply

Advisor sugere melhorias em 5 categorias mas não aplica sozinho. Você precisa agir sobre cada recomendação. Não confunda com Defender for Cloud (foco em segurança).

Advisor suggests improvements in 5 categories but doesn't apply them itself. You need to act on each recommendation. Don't confuse with Defender for Cloud (security focus).
